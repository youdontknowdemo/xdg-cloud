# Implementation Plan: TUI Offload Manager (`xdg-tui`)

> Generated by `/PACT:plan-mode` on 2026-07-03
> Status: IMPLEMENTED

## Summary

Build a Python (stdlib-first) TUI dashboard that renders every registry entry from
`bin/cloud-xdg-provision.sh` with per-entry state, and drives the script's lanes interactively:
offload / hydrate for the 3 CODE containers, plus reclaim and iCloud
(status/download/evict), with classify/offload-status as the read-only backbone. The TUI is a
**strict wrapper** — every action is a normal per-invocation run of the provision script
(dry-run preview → explicit confirm → `--apply`), so the script's guards (lock, not-root,
degenerate-path refusal, master trap, read-back verify) remain the sole enforcement layer.
Slice 0 adds a versioned `--porcelain` output mode to the two read-only commands so the TUI
never parses human-oriented output.

**User decisions already made (2026-07-03):**
1. **Scope**: registry dashboard (all classify rows shown; CODE rows action-enabled). NOT a
   sub-repo file browser — `resolve_code_target` refuses anything outside the 3 `CODE_KEYS`
   (`bin/lib/xdg-common.sh:159`); per-subdir offload would be a separate future
   provision-script feature.
2. **Stack**: Python. (Diverges from 3-of-4 specialist recommendation of pure bash 3.2 —
   recorded as an accepted trade-off; consequences in Risks + Cross-Cutting.)
3. **v1 lanes**: all data-movement lanes — offload, hydrate, reclaim, iCloud
   (status/download/evict) — plus classify/offload-status views.

**Orchestrator interpretation (flagged for user confirmation at approval):** "all lanes" is
read as all *file-management* lanes. The dotfiles lane and the default whole-home
provision/symlink lane are **excluded** from the TUI: they are setup/config operations, not
file management, and the provision lane behind a menu key is the highest-blast-radius action
in the toolkit.

**Effort**: Medium

#### Components Affected

|-----------|-------------|--------|

| `bin/xdg-tui.py` (new) | New | Python TUI: pure core (parser, action planner, result interpreter) + injectable executor + curses render/input loop. |
| `bin/xdg-tui` (new) | New | Tiny bash-3.2 launcher: python3 detection with graceful failure message, then `exec python3 "$dir/xdg-tui.py" "$@"`. Keeps the `bin/*.sh`-adjacent UX. |
| `Makefile` | Modify | `lint`/`test` gain python steps with graceful runtime-absent skip. |
| `tests/smoke.sh` | Modify | New porcelain golden-format group (bash-side contract tests). |
| `tests/tui/` (new) | New | Python unit/integration tests (stdlib `unittest` unless PREPARE rules otherwise). |
| `docs/architecture/xdg-cloud-adr.md` | Modify | Amend Decision 3 (no-python) with the TUI exception + rationale. |

#### Design Approach
Separate executable, never a `--tui` mode (the script's one-lane-per-invocation `set_mode`,
per-invocation lock, and per-invocation trap all argue for it; each TUI action is then a
normal fully-guarded run). Three-layer seam (test-engineer hard requirement):

1. **Pure core** (no tty, no subprocess): porcelain → entry model; (entry, op, confirmed) →
   exact argv; (rc, out, err) → UI state. `plan_action(entry, op, confirmed=False)` MUST
   omit `--apply`.
2. **Executor** (thin, injectable): runs argv; for read-only calls captures stdout+stderr
   fully (drain completely — never close the pipe early, SIGPIPE-141 hazard); for `--apply`
   calls does terminal handover: leave curses (`endwin`), run child attached to the real
   tty (rclone `--progress`, guard refusals, and the script's trap output render natively),
   record rc, "exit N — press Enter", re-init curses, refresh status.
3. **Render/input loop**: consumes the model, emits intents. Nothing else.

#### Key Decisions
| Decision | Options | Recommendation | Rationale |
|----------|---------|----------------|-----------|
| Process boundary | `--tui` mode vs separate executable | **Separate executable** | one-lane invariant, per-invocation lock/trap, 2189-line script vs 500-600 guidance |
| Data contract | parse human output / read state files / `--porcelain` | **`--porcelain`** | human format is drift-prone; state files are an internal contract; porcelain is small, versioned, golden-tested |
| Porcelain schema | pipe vs TSV | **Pipe-delimited**, version header `porcelain=1`, schema `class\|canonical\|localName\|state\|remote\|git` with fixed state enums (`local\|offloaded\|absent\|symlink`), extend by appending fields only | matches the repo's registry + `field()` idiom (`xdg-common.sh:29,147-155`); documented assumption: no `\|` in keys/remotes |
| Python UI layer | textual (pip) vs stdlib curses | **stdlib curses** (pending PREPARE go/no-go) | zero pip deps preserves what's left of the zero-install story |
| Mutation flow | direct `--apply` vs two-invocation | **dry-run preview → typed confirm → `--apply`** | preserves the repo's dry-run-default posture in the UX; TUI adds no validation of its own |
| iCloud evict consent | single-key vs typed acknowledgment | **Typed acknowledgment** (e.g., type the path or "evict" back) before the TUI passes `--i-understand-data-loss-risk` | the flag is deliberate consent friction; the TUI must preserve equivalent friction (architect ETHICS-adjacent concern, mitigated per user's all-lanes decision) |
| Concurrency | background jobs vs foreground | **Foreground, one op at a time** | Ctrl-C reaches the child directly; mkdir lock makes serial the only honest mode |
| Failure handling | interpret stderr vs relay | **Relay verbatim, refresh status after every action** (rc 130 = user cancel) | keeps the wrapper honest; the script's messages ARE the guard surface |

#### Interface Contracts
- **Invocation matrix** (TUI always addresses entries by canonical key, never local name):
  refresh = `--offload-status --porcelain` / `--classify --porcelain`; plan = `--offload <key>`
  (dry-run default); apply = same + `--apply`; hydrate analogous; reclaim =
  `--reclaim <path>` then `--reclaim <path> --apply`; iCloud = `--icloud-status/-download/-evict <path>`
  (+ typed-consent gate before `--i-understand-data-loss-risk`).
- **Exit codes**: 0 / 1 / 130 only; no new taxonomy requested.
- If dry-run output contains `WOULD BLOCK`, suppress the confirm prompt (UX nicety, not policy).

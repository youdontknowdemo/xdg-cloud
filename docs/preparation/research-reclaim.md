# Research: `--reclaim` safe-pattern set + detection heuristics

> PACT Prepare-phase research for the `--reclaim` mode (Task #90) — a build-artifact
> sweep that **DELETES** regenerable junk to free disk. **Analysis only.** The crux is
> **anti-data-loss detection**: a rule set that guarantees we never delete a real dir.
> Matches repo conventions (bash 3.2, dry-run default + `--apply` gate, `run()`
> discipline, symlink/degenerate-path guards).

## Executive Summary

`--reclaim` is the delete-side counterpart to offload: it removes **regenerable**
build artifacts (Rust `target/`, `node_modules`, Gradle/Maven build output,
`__pycache__`, and a fixed allow-list of global caches). Because it deletes, the entire
design rests on **false-positive-safe detection**. The governing rule:

> **A candidate is reclaimable only if it is ANCHORED by a sibling toolchain manifest
> proving it is build output, AND (for any generic/ambiguous name) it is git-IGNORED /
> untracked. Anything TRACKED is never touched. Generic names (`build`/`dist`/`out`)
> with no manifest → NEVER reclaim. When in doubt → EXCLUDE and document.**

Prefer **tool-native clean** (`cargo clean`, `go clean`, `npm cache clean`, `brew
cleanup`, `pip cache purge`) over `rm -rf` when the toolchain is present — the tool
knows what is safe. `rm -rf` is the guarded fallback. **Never a blind `$HOME` tree
walk**: per-project artifacts are found by walking *within* an explicit root; global
caches are a **separate fixed allow-list** only swept when explicitly requested.

## Reasoning Chain

1. Motivation (real machine): disk = regenerable build junk (`target/`, `.gradle`,
   npm) + a VM; deleting build junk freed ~3 GB where cloud-offload freed ~0. So the
   feature has real value **iff** it's safe.
2. Deleting is irreversible → detection must have **zero false positives** on real dirs.
3. Two independent signals establish "this is build output": (i) an **anchor** — a
   sibling manifest the toolchain owns; (ii) **git-ignored/untracked** — the project
   itself declares it disposable. Requiring both (for ambiguous names) makes a false
   positive require *two* independent coincidences.
4. Generic names (`build`/`dist`/`out`) are real source-dir names too → anchor alone
   is insufficient → require gitignored, and refuse outside a repo.
5. Tool-native clean is safer than `rm` (respects in-progress state, knows layout) →
   prefer it; `rm` only as guarded fallback.
6. A blind `$HOME` walk risks matching precious dirs by name → constrain to an explicit
   root + a fixed global-cache allow-list.

## 1. The curated pattern table

Legend — **Verdict**: ✅ include (default) · ⚠️ opt-in only · ❌ exclude.
**Scope**: *project* (found by walking within root) · *global* (fixed allow-list).

### Per-project artifacts (walk within root)

| Pattern (dir) | Anchor / detection rule | Reclaim method | Verdict |
|---|---|---|---|
| `target/` (Rust) | sibling `Cargo.toml` | `cargo clean` in project dir if `cargo` present, else guarded `rm -rf target` | ✅ |
| `target/` (Maven) | sibling `pom.xml` | `mvn clean` if `mvn` present, else guarded `rm -rf target` | ✅ |
| `node_modules/` | sibling `package.json` AND no `.git` **inside** node_modules (that = a checked-out dep) | guarded `rm -rf` (no cache-safe tool clean for the folder itself; regenerated by `npm/pnpm/yarn install`) | ✅ |
| `.next/` `.nuxt/` `.svelte-kit/` `.turbo/` | sibling `package.json` | guarded `rm -rf` | ✅ (names are framework-specific/unambiguous) |
| `dist/` `build/` `out/` (JS) | sibling `package.json` **AND** git-ignored/untracked | guarded `rm -rf` | ⚠️ require BOTH; ❌ if not gitignored or outside a repo |
| `__pycache__/` `*.pyc` | name is unambiguous (Python bytecode) | guarded `rm -rf` / `find -name '__pycache__' -type d` | ✅ |
| `.pytest_cache/` `.mypy_cache/` `.ruff_cache/` `.tox/` | unambiguous tool-owned name | tool-native where trivial (e.g. `tox` recreates); else guarded `rm -rf` | ✅ |
| `*.egg-info/` | unambiguous setuptools metadata | guarded `rm -rf` | ✅ |
| `build/` `dist/` (Python setuptools) | sibling `pyproject.toml`/`setup.py` **AND** git-ignored | guarded `rm -rf` | ⚠️ require BOTH; ❌ otherwise (generic name) |
| `build/` (Gradle) | sibling `build.gradle` or `build.gradle.kts` **AND** git-ignored | `gradle clean`/`./gradlew clean` if present, else guarded `rm -rf` | ⚠️ require BOTH (generic name) |
| `build/` + `CMakeCache.txt` + `CMakeFiles/` (CMake) | sibling `CMakeLists.txt` **AND** the `CMakeCache.txt`+`CMakeFiles/` markers inside | guarded `rm -rf build` (out-of-source build dir) | ⚠️ require the CMake markers (proves it's a CMake build dir, not a source `build/`) |
| `.venv/` `venv/` | sibling `pyproject.toml`/`requirements.txt` + `pyvenv.cfg` inside | (recreate via `python -m venv` + install) | ❌ default (opt-in only — holds pinned/editable installs; see §5) |
| `vendor/` (Go/PHP) | — | — | ❌ (can be committed; often tracked) |

### Global caches (fixed allow-list — only when explicitly requested / root == `$HOME`)

| Cache | Location | Reclaim method | Verdict |
|---|---|---|---|
| Homebrew downloads | `$(brew --cache)` | `brew cleanup [--prune=all]` | ✅ (opt-in global) |
| npm cache | `~/.npm/_cacache` | `npm cache clean --force` (self-healing; `npm cache verify`) | ✅ opt-in |
| pip wheel cache | macOS `~/Library/Caches/pip`, Linux `~/.cache/pip` | `pip cache purge` | ✅ opt-in |
| Go build/test cache | `~/Library/Caches/go-build` (macOS) / `$GOCACHE` | `go clean -cache` / `go clean -testcache` | ✅ opt-in (user-level, NOT per-project) |
| Gradle cache | `~/.gradle/caches` | guarded `rm -rf` (no safe partial tool clean; re-downloaded) | ⚠️ opt-in (large; fully regenerable but re-downloads) |
| Xcode DerivedData | `~/Library/Developer/Xcode/DerivedData` | guarded `rm -rf` (rebuilt on next build) | ✅ opt-in |
| Xcode iOS DeviceSupport / Archives | `~/Library/Developer/Xcode/{iOS DeviceSupport,Archives}` | guarded `rm -rf` | ⚠️ opt-in (Archives may hold shippable builds — DEFAULT EXCLUDE Archives) |
| Go module cache | `$GOMODCACHE` / `~/go/pkg/mod` | `go clean -modcache` | ⚠️ opt-in (read-only downloaded deps; slow to re-fetch) |
| Cargo registry cache | `~/.cargo/registry/cache` | guarded `rm -rf` | ⚠️ opt-in (download cache; re-fetched) |
| Maven local repo | `~/.m2/repository` | guarded `rm -rf` | ❌ default (download cache; slow re-fetch; can hold locally-installed artifacts) |
| XDG cache | `~/.cache/*` | — | ❌ blanket (too broad; only sweep specific known subdirs, never the whole tree) |

> **`go clean -cache/-testcache/-modcache` and `pip cache purge`/`npm cache clean` are
> USER-GLOBAL, not per-project** — they belong to the global allow-list, not the
> in-root walk. Flag this to the architect: running them is a global side effect.

## 2. The detection-safety model (the load-bearing rule)

**(a) Anchor-by-manifest alone, or ALSO require git-ignored?**
Recommend a **tiered rule**:
- **Unambiguous artifact names** (`target/`+`Cargo.toml`, `node_modules/`+`package.json`,
  `__pycache__`, `.pytest_cache`, `.mypy_cache`, `.ruff_cache`, `.next`, `.nuxt`,
  `.svelte-kit`, `.turbo`, `*.egg-info`): the name itself is a strong signal → **anchor
  is sufficient**. Additionally verify git-ignored **when inside a repo** (belt-and-
  suspenders; they're conventionally ignored anyway) but don't hard-require it outside.
- **Generic/ambiguous names** (`build/`, `dist/`, `out/`): **REQUIRE anchor AND
  git-ignored/untracked.** A tracked `build/` with real content is thereby never touched.
  Detection: `git check-ignore -q "$dir"` (exit 0 = ignored) — the canonical git test.
  Cross-check `git ls-files --error-unmatch` returns non-zero (untracked).

**(b) Dir OUTSIDE any git repo (no gitignore context):** — tightened per team-lead
review to be maximally conservative for a DELETE feature. Three sub-tiers:
- **Tool-native-authoritative anchors** (`target/`+`Cargo.toml`→`cargo clean`;
  `target/`+`pom.xml`→`mvn clean`; Gradle `build/`+`build.gradle`→`gradlew clean`):
  **anchor-alone is acceptable.** Rationale: the toolchain itself bounds the deletion —
  `cargo clean` removes only that package's `target/` and nothing else — so we are not
  relying on our own path logic to decide what dies. This is the ONLY case where
  anchor-alone is trusted without a git context.
- **Pure-bytecode/tool-cache names** (`__pycache__/`, `*.pyc`, `.pytest_cache/`,
  `.mypy_cache/`, `.ruff_cache/`, `*.egg-info/`): name is intrinsically unambiguous and
  the content is trivially regenerable → anchor/name-alone acceptable (low-risk even via
  `rm`).
- **Everything else outside a repo** — including `rm`-based unambiguous names like
  `node_modules/`, `.next/`, and ALL generic names (`build/dist/out`): **REFUSE.**
  Without a git context AND without a tool-native clean to bound the deletion, we can't
  prove disposability strongly enough to `rm`. Exclude and log the skip.

  *(This is stricter than "any unambiguous name is fine outside a repo": a lone
  `node_modules` in `/tmp` is almost certainly disposable, but "almost certainly" is not
  the bar for an irreversible delete — so it's refused unless inside a repo where
  `git check-ignore` confirms it.)*

**(c) Generic-named dirs with NO manifest → NEVER reclaim.** Hard exclude.

**(d) Traversal guards (non-negotiable):**
- **Never follow symlinks** — enumerate with `find "$root" -type d` (no `-L`); skip any
  path component that is a symlink. A symlinked `node_modules` → real code elsewhere.
- **Never ascend above the root** — realpath-normalize `$root` and every candidate;
  refuse any candidate whose resolved path doesn't have `$root` as a prefix.
- **Degenerate-path guard before every delete** — refuse if the target resolves to `/`,
  `$HOME` itself, empty/unset, or `$root` itself; refuse `rm -rf` on a path with `..`
  after normalization. (Mirror cloud-xdg's existing defensive posture.)
- **Stop descending into a matched dir** — once `node_modules/` is matched, don't walk
  inside it (avoids nested-match churn and the checked-out-dep trap).

## 3. Scope & default root

**Recommendation:** default `--reclaim` operates on the **explicit root arg**; if none
given, default to a **narrow dev root** (the current working directory, or `~/repos` if
it exists) — **never a blind `$HOME` walk**.

- **Per-project artifacts** (`target/`, `node_modules`, Gradle/CMake `build/`,
  `__pycache__`) → found by **walking within the root**, applying the §2 detection rule.
- **Global caches** (`~/.gradle/caches`, `~/.npm/_cacache`, Xcode DerivedData, `brew`
  cache, etc.) → a **separate FIXED allow-list**, swept **only** when explicitly opted
  in (e.g. `--global`/`--caches`) or when the root is exactly `$HOME`. They are **never**
  discovered by tree-walking.
- A `--root PATH` narrows the per-project walk; global caches remain gated behind the
  explicit opt-in regardless of root.

This gives the ~3 GB win (project build junk under `~/repos`) with no risk of a stray
`$HOME` match, and makes the large, slower-to-regenerate global caches a deliberate
choice.

## 4. Tool-native vs `rm -rf`

**Prefer tool-native when the toolchain is present** — it knows the layout, respects
in-progress builds, and won't over-delete:

| Pattern | Tool-native (preferred) | `rm` fallback |
|---|---|---|
| Rust `target/` | `cargo clean` (in project dir) | `rm -rf target` (guarded) |
| Maven `target/` | `mvn clean` | `rm -rf target` |
| Gradle `build/` | `./gradlew clean` / `gradle clean` | `rm -rf build` |
| Go caches | `go clean -cache` / `-testcache` (global) | — |
| npm cache | `npm cache clean --force` | `rm -rf ~/.npm/_cacache` |
| pip cache | `pip cache purge` | `rm -rf <pip cache>` |
| Homebrew | `brew cleanup` | — |
| `node_modules`, `.next`, `__pycache__`, DerivedData | (no cache-safe tool clean) | guarded `rm -rf` |

Caveats to flag: (1) tool-native global commands (`go clean -modcache`, `npm cache
clean`) have **global** side effects — scope carefully and only under the opt-in.
(2) `cargo clean`/`gradle clean` must be run **in the project dir** so they only touch
that project. (3) `rm -rf` always passes through the §2(d) degenerate-path guard.

**Tool-native safety confirmation** (each command's blast radius is bounded — verified
against each tool's docs; this is *why* they're preferred over `rm`):

| Command | What it deletes (and nothing else) |
|---|---|
| `cargo clean` | Only the current package's `target/` (the configured target dir). Does NOT touch `~/.cargo`. |
| `mvn clean` | Only the module's `target/` (the `build/directory`). |
| `gradle clean` / `./gradlew clean` | Only the project's `build/` (the `clean` task deletes `$buildDir`). |
| `go clean -cache` / `-testcache` | Only the build cache / cached test results. `-modcache` (module cache) is separate and opt-in. |
| `npm cache clean --force` | Only `~/.npm/_cacache`; cache is self-healing (`npm cache verify`). |
| `pip cache purge` | Only pip's wheel/http cache directory. |
| `brew cleanup` | Only outdated downloads / stale lock files (not installed formulae). |

None of these deletes source, VCS state, or user data — the toolchain owns exactly the
path it removes. That bounded blast radius is the safety argument for preferring them,
and it underwrites the §2(b) "anchor-alone acceptable outside a repo" carve-out for the
tool-native-authoritative cases only.

## 5. Exclusions (explicit — never reclaim, and why)

- **`.venv/` `venv/` (default)** — recreatable in principle, but holds **pinned versions,
  editable/`-e` installs, and compiled extensions** that may not reproduce if
  `requirements`/lockfiles have drifted. Opt-in only (`--venvs`), never default.
- **`vendor/`** — Go/PHP vendored deps are frequently **committed** (a valid workflow);
  can be tracked. Exclude.
- **Generic `build/` `dist/` `out/` without a manifest** — indistinguishable from real
  source/output dirs. Exclude.
- **Anything TRACKED in git** (`git ls-files` matches) — the project treats it as source.
  Exclude unconditionally.
- **`node_modules/` containing a `.git`** — a checked-out dependency, not an `install`
  artifact. Exclude.
- **Git submodules** (paths in `.gitmodules`, or a `.git` *file* pointing into
  `.git/modules`) — deleting orphans the superproject's checkout. Exclude.
- **`~/.m2/repository`, Cargo registry, Go modcache (default)** — download caches;
  regenerable but slow/bandwidth-heavy re-fetch; opt-in only.
- **Xcode `Archives`** — may hold shippable/notarized builds; default exclude.
- **`.git/` itself, and any dir outside `$root`** — never.

If any additional candidate cannot be made false-positive-safe, the rule is: **EXCLUDE
and document** rather than risk a real dir.

## Deliverable for the architect

1. Implement the **tiered detection rule** (§2): unambiguous-name = anchor(+gitignored
   in-repo); generic-name = anchor AND gitignored, refuse outside a repo.
2. Two discovery paths: **in-root walk** for per-project artifacts; **fixed allow-list**
   for global caches behind an explicit opt-in. Never a blind `$HOME` walk.
3. **Tool-native clean preferred**, `rm -rf` guarded fallback with the §2(d) degenerate-
   path/symlink/no-ascend guards.
4. Dry-run by default (`--apply` to act), reporting reclaimable bytes per candidate —
   consistent with the repo's existing safe-by-default posture.

- **[LOW]** CMake in-source builds (no separate `build/`) — reclaim must NOT try to clean an in-source tree; the `CMakeCache.txt`+`CMakeFiles/` marker requirement handles this (an in-source build is excluded by needing a dedicated build dir).
